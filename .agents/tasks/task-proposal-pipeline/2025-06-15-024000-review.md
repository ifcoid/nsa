# Proposal ingest pipeline with citation validation and knowledge graph

The change adds a parallel pipeline for academic proposal writing alongside the existing SLR pipeline. It introduces a `ProposalModule` interface, a multi-state ingest module (P0) that walks sessions through BibTeX parsing, PDF upload, embedding, and Neo4j knowledge graph construction, a `CitationGate` that validates LaTeX citations against MongoDB and Qdrant, a BibTeX state-machine parser, seven new Mongo repository methods, nine HTTP endpoints behind the existing PASETO auth middleware, and an async orchestrator that mirrors the SLR pipeline's Execute/ExecuteAsync pattern.

**Watch for:** no per-user ownership check on sessions (any authenticated user can mutate any proposal session) [confirmed]; `UpsertProposalRefs` issues one `UpdateOne` per ref with no bulk write, creating an O(n) round-trip storm for large bib files [confirmed]; the `UploadPDF` handler doesn't actually store or process the file — it only flips a boolean [confirmed]; the semantic-verification step in `CitationGate` compares against a global Qdrant collection with no session scoping [confirmed].

## High-level view

The pipeline adopts the same prefix-routing design as the SLR pipeline: session status strings carry a module prefix (`P0_`), and the orchestrator dispatches to the matching module. Only P0 is registered, and there's no mechanism to chain into a subsequent proposal-writing module — the pipeline stops at `P0_DONE`.

The HTTP layer exposes a full CRUD surface for proposal sessions plus specialized endpoints for bib upload, PDF marking, and embed-endpoint registration. All endpoints sit behind `AuthMiddleware`, but none verify that the authenticated user owns the session being accessed. Any valid token grants access to any proposal session.

The citation gate performs two-phase validation — existence in Mongo then semantic similarity via Qdrant — but the semantic search queries a global collection without filtering by session or by the specific paper's DOI/cite_key. A high-similarity hit from an unrelated paper would pass the gate, and the gate only fires if something upstream calls `ValidateCitations` (nothing in the current diff does).

The BibTeX parser is a hand-rolled state machine that handles brace-depth tracking and the common field delimiters. It works for well-formed `.bib` exports from Zotero/Mendeley but will silently mis-parse entries with string concatenation (`#` operator), escaped braces inside quoted values, or `@string` macros used as field values.

The knowledge graph builder maps keywords to `Method` nodes unconditionally and labels author-to-paper edges as `CITES`, which is semantically incorrect — an authorship relationship is not a citation.

## Session access control gap

The `ProposalHandler` methods extract the session ID from the URL path and directly query MongoDB without comparing the session's `UserID` against the authenticated user. The auth middleware injects PASETO claims into the request context (visible in the existing `auth.go`), but none of the proposal handlers read them. A user who obtains a valid token for *their* account can enumerate session IDs and read, modify, or trigger any other user's proposal pipeline.

The fix is straightforward: after fetching the session, compare `session.UserID` to the claim in `req.Context()` and return 403 if they diverge.

## Batch-write performance in UpsertProposalRefs

```go
for i := range refs {
    refs[i].SessionID = sessionID
    filter := bson.M{"cite_key": refs[i].CiteKey, "session_id": sessionID}
    update := bson.M{"$set": refs[i]}
    opts := options.Update().SetUpsert(true)
    _, err := collection.UpdateOne(ctx, filter, update, opts)
    if err != nil { return err }
}
```

Each iteration issues a network round-trip to MongoDB. For a 200-entry bib file this is 200 sequential writes. The `mongo` driver supports `collection.BulkWrite` with `UpdateOneModel` in `options.BulkWrite().SetOrdered(false)`, which would batch these into a single wire command. This matters because the handler calls this synchronously in the HTTP request path — the user waits for all N upserts to complete before receiving a response.

## UploadPDF marks is_embedded without processing

The `UploadPDF` handler calls `req.FormFile("file")` and `ParseMultipartForm` but never reads the file content or stores it anywhere. It only flips `is_embedded: true` on the matching ref document. The file handle is opened and immediately closed. Either the handler should store the PDF (e.g. to object storage) before marking it embedded, or the endpoint name is misleading and should be renamed to something like `mark-pdf-embedded`.

This means the P0 state machine will transition out of `P0_WAITING_PDFS` once all refs are marked, but no actual PDF data exists in the system.

## Citation gate scoping and wiring

`CitationGate.ValidateCitations` calls the package-level `SemanticSearch(ctx, claimText, 5)` which queries the `scientific_articles` collection in Qdrant. There is no filter payload restricting results to the current session's papers. If a claim about "transformer attention" matches a high-scoring paper from a completely different session, the gate considers the citation valid.

Nothing in the diff calls `ValidateCitations`. The `P0Ingest` module never invokes it; the handlers don't invoke it; there's no `P1` module yet. The gate compiles and is architecturally ready, but it's dead code in the current implementation.

The semantic check logic also has a conceptual gap: it looks for *any* result with score >= 0.75 that has a non-empty DOI or Title, rather than checking whether the *specific cited paper* appears in the results. Two unrelated but semantically similar papers would satisfy the threshold.

## Knowledge graph edge semantics

In `buildKnowledgeGraph`, author-to-paper relationships use edge type `CITES`:

```go
edges = append(edges, repository.GraphEdge{
    Type:       "CITES",
    SourceNode: authorNode,
    TargetNode: paperNode,
    ...
})
```

An author doesn't cite their own paper — they *authored* it. The edge label is semantically wrong and will produce confusing graph queries downstream. Similarly, papers by the same author are linked with `EXTENDS`, which implies intellectual extension; "same_author" is a co-occurrence relationship, not a conceptual one.

Keywords are uniformly mapped to `Method` nodes regardless of content. A keyword like "systematic review" or "MNIST dataset" is not a method. Without classification (even a simple heuristic or LLM call), the graph will be noisy.

## BibTeX parser edge cases

The parser will fail on:

- `@string` macros used as field values (e.g., `journal = jmlr` where `jmlr` is defined in a `@string` block) — the bare-value branch captures it as a literal string "jmlr" rather than expanding the macro.
- Concatenation with `#` (e.g., `author = first # " and " # second`) — the parser will capture everything up to the next comma as a single value including the `#`.
- Escaped braces in quoted values (`title = "A {B} approach"`) — this works only when brace-delimited, not when quote-delimited, because the quote parser stops at the first `"` regardless of nested braces.

These are common in exports from BibDesk, JabRef, and manual `.bib` files. The failure mode is silent: fields will contain garbage rather than raising a parse error.

## Pipeline orchestrator: no external cancellation

`ExecuteAsync` spawns a goroutine with a 6-hour context timeout and loops until the session reaches a `WAITING`, `ERROR`, or `DONE` state. The single-worker guard (`activeWorkers` map) prevents duplicate goroutines, but there's no way to cancel a running pipeline from outside except by waiting for the 6-hour timeout or killing the process. The `activeCancelFuncs` map stores cancel functions but no endpoint or mechanism exposes them.
